# _return_type not renamed to __dataclass_return_type__

## Severity: MEDIUM

## Description

As part of gh-96151, CPython 3.12 renamed all internal variable names
in generated code to dunder-prefixed names to avoid name collisions with
user-defined field names. Most names had been updated correctly in the
backport:

- `_dflt_{name}` -> `__dataclass_dflt_{name}__`
- `_HAS_DEFAULT_FACTORY` -> `__dataclass_HAS_DEFAULT_FACTORY__`
- `_type_{name}` -> `__dataclass_type_{name}__`
- `object` -> `__dataclass_builtins_object__`

But `_return_type` in `_create_fn()` had **not** been renamed.

## Original code (dataclasses.py)

```python
if return_type is not MISSING:
    locals["_return_type"] = return_type
    return_annotation = "->_return_type"
```

## CPython 3.12 (reference)

```python
if return_type is not MISSING:
    locals['__dataclass_return_type__'] = return_type
    return_annotation = '->__dataclass_return_type__'
```

## Impact

A dataclass field named `_return_type` could collide with the internal
variable name in the generated `__init__` function's type annotation.

## Sources

- CPython 3.12 `Lib/dataclasses.py`, `_create_fn()`
- gh-96151: https://github.com/python/cpython/issues/96151

## Resolution

Fixed as part of the `_create_fn()` closure-wrapping update (issue 2).
The variable is now named `__dataclass_return_type__`, matching
CPython 3.12.
